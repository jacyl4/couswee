## 背景

couswee 已经具备 SvelteKit 前端、账号列表读取、用量记录读取和账号切换调用能力。本轮不是泛泛美化，而是按用户更新后的效果图重做界面契约：页面标题变为 `couswee 账号切换监测`，并新增账号管理工具条、账号复选框、选中计数、`新增账号` 与 `删除选中` 操作。

本设计继续沿用之前已确认的语义：`5h` 和 `Weekly` 展示的是剩余流量/剩余容量百分比，不是已消耗用量。真实数据仍来自本地 API；效果图中的账号名和百分比只作为视觉与状态示例。

## 目标 / 非目标

**目标：**
- 严格贴近新版效果图的单页结构：暗色背景、暖色文字、玻璃拟态卡片、顶部搜索、三张统计卡、账号管理工具条、可选中的账号卡片、进度条、切换按钮和底部同步提示。
- 将页面文案统一为中文文档与中文结构文案；效果图中明确为英文的指标标签（`5h limit`、`Weekly limit`、`left`、`resets`）保持英文以匹配图面。
- 支持账号筛选、选中/取消选中、选中计数、删除选中入口、可用/建议切换/最后同步概览计算。
- 通过确定性阈值规则，从真实剩余百分比推导绿色、黄色、红色视觉状态。
- 保持本地优先与轻依赖实现，优先复用现有 SvelteKit 和后端接口。
- 让本 change 成为下一次界面实现的权威目标，旧 change 中冲突的 UI 描述不再优先。

**非目标：**
- 不重做 Codex 凭据切换底层机制。
- 不重做 Codex 用量采集来源，除非新版界面无法从现有字段拿到必须信息。
- 不引入大型 UI 组件库、图表库、Electron/Tauri 壳或外部服务。
- 不强制生产数据使用效果图里的示例账号名。
- 不强制实现真实 `⌘K` 全局快捷键；视觉提示必须有，实际快捷聚焦可作为低风险增强。
- 不扩展复杂账号管理模型；新增/删除账号只补最小可用本地接口，字段沿用现有账号 JSON 契约。

## 决策

### 1. 用新版效果图覆盖旧 UI 契约
本 change 直接把新版效果图作为验收标准。旧 change 中的“表格列表”“通用刷新按钮”“单纯账号列表”等描述，如果与新版图冲突，应在实现时删除或降级。

备选方案：继续在旧 UI 上局部加管理按钮。拒绝原因：新版图已经改变了页面信息架构，仅局部修补会继续留下表格化和泛后台感。

### 2. 账号卡片需要内置选择列
每张账号卡片左侧增加复选框，选中后卡片边框变为绿色高亮，并在账号管理工具条右侧显示 `已选择 N 项`。这让 `删除选中` 有明确对象，也与新版图中 `backup-01` 被选中的状态一致。

备选方案：只在账号管理区域提供删除按钮，不显示选择态。拒绝原因：用户无法知道删除目标，且不符合新版效果图。

### 3. 账号管理工具条独立于账号卡片
`账号管理` 以一个折叠式/分组式面板呈现在顶部概览和账号卡片之间，包含齿轮图标、标题、展开箭头、`新增账号`、`删除选中`、已选择计数。实现上可以先让折叠按钮控制工具条内容显示/隐藏；如果为了匹配首版图面，也可以默认展开。

备选方案：把新增/删除按钮放进每张卡片。拒绝原因：新版图把管理操作设计为列表级操作，不是单卡片操作。

### 4. 状态由剩余百分比确定
前端以 5h 和 weekly 两个剩余百分比的较低值作为状态基准：
- `ok`：最低剩余值大于 50%，绿色，徽标 `可用`。
- `warn`：最低剩余值在 10% 到 50% 之间（含边界），黄色，徽标 `接近用尽`。
- `danger`：最低剩余值小于 10%，红色，徽标 `冷却中`。

`建议切换` 统计 `warn` 账号数；`danger` 表示冷却中，默认不作为推荐切换目标。若后端未来提供更权威的状态字段，可以在不破坏该规则的前提下合并使用。

备选方案：完全依赖后端状态字符串。拒绝原因：当前 UI 必须稳定复现效果图中的颜色与状态，而后端字段不一定完整。

### 5. 文案策略：文档中文，图面混合文案按图保留
OpenSpec 文档全部使用中文；界面结构文案也使用中文。但效果图中的指标行明确使用 `5h limit`、`Weekly limit`、`left`、`resets`，实现时应按图保留，避免为了全中文而破坏视觉一致性。

备选方案：把所有界面文案完全汉化。拒绝原因：用户要求严格按效果图。

### 6. CSS 原生实现玻璃拟态
视觉系统用 CSS 变量、径向渐变、半透明边框、阴影、柔光、卡片背景、原生进度条/条形元素实现，不新增依赖。

备选方案：引入 UI/图表库。拒绝原因：目标界面不需要额外依赖，且用户偏好不要多装。

### 7. 同步仍采用轮询
沿用当前自动刷新方式，前端每分钟刷新用量并更新 `最后同步`；底部显示 `数据每分钟自动同步`。手动刷新不是新版图主元素，可以不放在首屏主要位置。

备选方案：WebSocket 或服务端推送。拒绝原因：增加后端复杂度，和本次 UI 目标无关。

## 风险 / 权衡

- [风险] 新版效果图在对话中提供，不一定已保存为仓库文件 → 缓解：文档明确写出新版图的结构、文案和状态；实现时以用户本轮图片为准。
- [风险] 真实账号名称、reset 文案长度不同 → 缓解：用弹性布局、最小宽度和必要的文本换行，优先保证核心百分比与按钮不被遮挡。
- [风险] 新增/删除账号接口过度扩展 → 缓解：只补 `POST /api/accounts` 与 `DELETE /api/accounts` 的最小本地 registry 操作，并用测试覆盖。
- [风险] 用量数据缺失时颜色判断不稳定 → 缓解：缺失值按 0 或未知安全处理，并显示错误/陈旧提示，不把已用量误标成剩余。
- [风险] 旧 change 存在冲突 UI 描述 → 缓解：本 change 在 proposal/spec/tasks 中明确为新版 UI 权威来源。
- [风险] 视觉匹配难以由自动测试完全证明 → 缓解：除构建和测试外，任务要求人工对照新版效果图进行视觉检查。
